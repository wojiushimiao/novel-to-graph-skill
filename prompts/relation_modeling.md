# 关系建模规则 — 轴关系模型

> S6 整合清洗阶段必读。基于轴索引的扇入优先模型。
> v0.4.0: T_main/T_branch 语义重构；K 维度移除；S_topo 双向；关系边仅作索引。
> v0.5.0: 新增 evolves_to 角色演化轴（§2.5b）；多档案并行角色 id 格式。
> v0.5.1: §9 S6 流程新增步骤 2（骨架审查与重做）和步骤 11（人物小传构建）；§人物小传构建指导。

---

## 1. 节点单元结构

每个节点单元 = 关键字 - 值（详细信息，可多条目） - 关系（关键字形式指向其他节点单元）

```
节点单元 = {
    关键字: entity_id (如 C_莫凡, E_module_博城篇)
    值:     detail_info (多条目详细信息, 500-1500字语义提炼)
    关系:   通过轴索引指向其他节点（优先扇入，不显示记录）
}
```

**信息架构原则（v0.4.0）**：节点详情内聚于 info 字段，关系边仅作索引（详见 §6）。

## 2. 轴关系类型（7种）

### 2.1 T_main — 时序主轴（纯索引）

| 项 | 规范 |
|----|------|
| **语义** | 剧情模块按时序串联，构成全局时序骨架（纯索引） |
| **方向** | E_module_A → E_module_B（时间先后） |
| **特殊性** | 独立扇出关系，仅连接剧情模块（E_module）实体 |
| **约束** | T_main 仅存在于 E_module 之间，禁止连接其他实体类型 |
| **强制性** | 每个 E_module 至少存在一条 T_main 出边或入边 |
| **示例** | E_module_博城篇 → E_module_入学篇 → E_module_博城保卫战 |

**纯索引说明（v0.4.0 语义重构）**：
- T_main 不再串联具体事件，仅作为剧情模块间的时序索引
- 具体事件通过 T_branch 挂载到剧情模块
- 避免 v0.3.0 中 T_main 被 5,938 条事件串联淹没的问题

### 2.2 T_branch — 时序分支轴

| 项 | 规范 |
|----|------|
| **语义** | 剧情模块内的具体事件组织，挂载到所属模块 |
| **方向** | E_module → E_event（模块指向事件） |
| **结构** | 每个剧情模块通过 T_branch 挂载其范围内的跨章节事件 |
| **扇出限制** | 每模块 T_branch 出边 ≤ 20 |
| **示例** | E_module_博城篇 → {E_觉醒仪式, E_博城保卫战, E_入学考核} |

**T_branch 挂载规则**：
- 每个跨章节事件单元必须挂载到其所属的剧情模块
- 事件通过 T_branch 建立与模块的归属关系
- T_branch 不再用于鱼骨图分解，改为模块→事件的直接挂载

### 2.3 S_topo — 空间拓扑轴（双向·v0.4.0 微调）

| 项 | 规范 |
|----|------|
| **语义** | 空间关系的拓扑归属，双向语义 |
| **方向（入边）** | 实体 → 地点（地理归属） |
| **方向（出边）** | 地点 → E_module/E_event（空间索引） |
| **示例（入边）** | C_莫凡 → L_博城; E_觉醒仪式 → L_魔法学院 |
| **示例（出边）** | L_博城 → E_module_博城篇; L_魔法学院 → E_觉醒仪式 |

**S_topo 双向语义说明（v0.4.0 微调新增）**：

- **入边（实体→L）**：实体归属到主场景，用于实体的空间定位
  - 实体出边 ≤ 3（一个实体最多归属 3 个地点）
  - 所有涉及地理位置的实体/事件都应挂载到 S_topo 入边
- **出边（L→E_module/E_event）**：地点指向其下发生的剧情单元，用于从地点出发独立遍历
  - 地点出边 ≤ 20（一个地点最多索引 20 个剧情单元）
  - 空间关系从辅轴提升为可独立遍历的平行主轴
- **双向冗余控制**：入边和出边各自独立建立，不要求成对存在
  - 同一对(实体, L)和(L, 剧情单元)语义不同，各自独立

### 2.4 A_causal — 因果辅轴

| 项 | 规范 |
|----|------|
| **语义** | 事件间因果关系，协助主轴构建完整关系网络 |
| **方向** | 原因事件 → 结果事件 |
| **约束** | 仅因果分叉点建关系（非线性连续） |
| **扇出限制** | 事件出边 ≤ 3 |
| **示例** | E_觉醒仪式 → E_同学嫉妒（因果分叉） |

### 2.5 A_arc — 角色弧光辅轴

| 项 | 规范 |
|----|------|
| **语义** | 角色成长轨迹里程碑，协助主轴构建完整关系网络 |
| **方向** | 旧阶段角色 → 新阶段角色 |
| **约束** | 仅弧光转折点建关系（非每次微小成长） |
| **扇出限制** | 角色出边 ≤ 5 |
| **示例** | C_莫凡_魔法学徒 → C_莫凡_双系法师 |

### 2.5b evolves_to（角色演化轴，v0.5.0 新增）

**定义**: 连接同一角色在不同 T_main 卷的档案，形成人物弧光链。

**触发条件**: 同一角色在多个 T_main 卷中出现，每个卷有独立的角色档案（id 格式: `C_{name}__T_main_vol_{idx}`）。

**关系方向**: 按卷索引递增方向（小→大），不可逆。

**示例**:
```
C_莫凡__T_main_vol_0  --evolves_to-->  C_莫凡__T_main_vol_1
C_莫凡__T_main_vol_1  --evolves_to-->  C_莫凡__T_main_vol_2
```

**描述格式**: "角色弧光: {name} 从第{i}卷到第{j}卷的演化"

**强度**: 固定为 strong

### 2.6 R_strong — 强关联索引

| 项 | 规范 |
|----|------|
| **语义** | 剧情模块通过关系索引组织内部实体（E_module→C/L/I/R），以及实体间的强关联 |
| **方向** | E_module → C/L/I/R（模块索引实体）; 实体 → 实体（强关联） |
| **适用场景** | 剧情模块实体集群、无明确时序的个人支线、个人专属武器功法、强关联个体（师徒/父子） |
| **扇出限制** | 每模块 R_strong 出边 ≤ 30 |
| **示例** | E_module_博城篇 → C_莫凡（模块索引）; C_莫凡 → I_雷印（专属武器）; C_莫凡 → C_张小候（挚友） |

## 3. K 维度移除说明（v0.4.0）

**K 维度（关联主题）已废弃**：

- v0.3.0 中 K 维度定义为关联主题，但实测变为散乱关键词列表（3,038 个散乱 T 值）
- v0.4.0 移除 K 维度，主题分类功能由剧情模块（E_module）承担
- 通过 R_strong 关系索引组织实体到剧情模块

**移除影响**：
- 坐标从六维（T/L/C/E/K/R）简化为五维（T/L/C/E/R）
- 主题归属通过 T 坐标（所属剧情模块 ID）+ R_strong 关系边共同表达
- 不再有独立的主题坐标轴

## 4. 传导路线定义

信息从全局骨架传导到叶节点的完整路线：

```
T_main 时序主轴（纯索引）
  │  E_module → E_module
  │  构成全局时序骨架
  ▼
T_branch 时序分支
  │  E_module → E_event
  │  剧情模块内的事件组织
  ▼
剧情模块信息集群（E_module + R_strong）
  │  剧情模块通过关系索引组织内部实体
  │  E_module + R_strong → C/L/I/R
  │  （注：原K维度已移除，主题分类功能由剧情模块承担）
  ▼
实体档案（被索引的叶节点）
  │  每个实体有唯一的五维坐标定位
  │  T=所属模块, L=主场景, C=主体角色, E=所属事件, R=规则子类
```

**传导层次**：
1. **L1 全局时序层**（T_main）：剧情模块间的全局骨架
2. **L2 局部时序层**（T_branch）：模块内事件的组织
3. **L3 实体索引层**（R_strong）：模块对内部实体的索引集群
4. **L4 叶节点层**：被索引的实体档案，含五维坐标定位

**辅助索引**：
- A_arc（角色弧光）：跨模块角色成长，C→C
- A_causal（因果关系）：事件因果分叉点，E→E
- S_topo（空间主轴·双向）：实体地理归属 + 地点空间索引

## 5. 扇出/扇入规则

### 5.1 核心原则

- **扇入优先**：实体尽可能仅扇入（被指向），不限制扇入数量
- **少扇出**：实体少扇出（指向其他），而是以主轴、辅轴作为索引进行扇入
- **例外**：剧情模块通过 R_strong 索引内部实体；强关联实体可直接扇出（R_strong）

### 5.2 出边（扇出）上限详表（v0.4.0 微调更新）

| 关系类型 | 连接对象 | 扇出上限 | 说明 |
|---------|---------|---------|------|
| T_main | E_module → E_module | 无限制 | 全局时序骨架 |
| T_branch | E_module → E_event | 每模块 ≤ 20 | 模块内事件组织 |
| R_strong | E_module → C/L/I/R | 每模块 ≤ 30 | 实体集群索引 |
| A_arc | C → C | 角色出边 ≤ 5 | 弧光追踪 |
| A_causal | E → E | 事件出边 ≤ 3 | 因果导航 |
| S_topo（入） | 实体 → L | 实体出边 ≤ 3 | 实体地理归属 |
| S_topo（出） | L → E_module/E_event | 地点出边 ≤ 20 | 空间索引（微调新增） |

### 5.3 实体类型出边上限

| 实体类型 | 出边上限 | 允许的扇出关系类型 |
|---------|----------|-----------------|
| 剧情模块 (E_module) | T_main 无限制 + T_branch ≤20 + R_strong ≤30 | T_main, T_branch, R_strong |
| 角色 (character) | ≤ 5 | R_strong, A_arc |
| 事件 (E_event) | ≤ 3 | A_causal |
| 地点 (location) | ≤ 20 (仅 S_topo 出边) | S_topo（出边: L→E_module/E_event） |
| 物品 (item) | 0 | （仅扇入） |
| 规则 (rule) | 0 | （仅扇入） |
| 系统 (system) | 0 | （仅扇入） |

### 5.4 入边（扇入）不受限制

所有实体类型的扇入数量不受限制，因为扇入不产生上下文负担。

## 6. 关系边仅作索引原则（v0.4.0 微调新增）

### 6.1 核心原则

**关系边是"指针+门控"，不是"内容容器"**。

- 关系边的职责：指向目标实体（target_id）+ 标注关系类型（relation_type）+ 简短说明（description）
- 关系边不承载详细内容：起因经过结果、人物背景、事件详情、规则定义等必须内聚在实体 info 字段中

### 6.2 关系 description 字段契约

| 约束 | 说明 |
|------|------|
| **推荐字数** | ≤ 50 字一句话说明 |
| **上限字数** | ≤ 100 字（硬限制，超过拒绝入库） |
| **空值允许** | 可为空字符串（仅作指针） |
| **禁止内容** | 起因经过结果、人物背景、事件详情、规则定义 |

### 6.3 正确与错误示例

```
# 正确：详情内聚于实体 info，关系边仅作索引
Entity(C_莫凡):
  info: "莫凡是天澜魔法高中的学生，觉醒雷系和光系...(500-1500字详情)"

Relation(C_莫凡 → C_张小候, R_strong):
  description: "同班同学与挚友"  # 一句话说明

# 错误：详情分散到关系边
Relation(C_莫凡 → C_张小候, R_strong):
  description: "莫凡与张小候在入学时相识，曾一起经历博城之难...(200字详情)"
```

### 6.4 校验规则

- 关系 description 字数 > 50 字：标记为疑似详情分散（E_REL_DESC_SUSPICIOUS），触发 LLM 重新归纳
- 关系 description 字数 > 100 字：拒绝入库（E_REL_DESC_TOO_LONG），强制 LLM 重新抽取
- 实体 info 字数 < 500 字 且 出边数 ≥ 3：标记为疑似详情分散（E_INFO_NOT_COHESIVE）

## 7. 轴索引化规则

### 7.1 主轴索引

剧情模块通过 T_main 串联，构成时序主轴。具体事件通过 T_branch 挂载到剧情模块。

```
时序主轴: E_module_博城篇 →(T_main)→ E_module_入学篇 →(T_main)→ E_module_保卫战篇
                    |                        |                        |
              (T_branch)              (T_branch)               (T_branch)
                    ↓                        ↓                        ↓
        E_觉醒仪式, E_入学考核      E_学院日常, E_修炼突破     E_博城保卫战, E_妖兽入侵
```

### 7.2 辅轴索引

辅轴（S_topo, A_causal, A_arc）作为索引层，实体通过辅轴进行扇入：

```
空间拓扑轴（双向）:
  入边: C_莫凡 →(S_topo)→ L_博城          # 实体地理归属
  出边: L_博城 →(S_topo)→ E_module_博城篇  # 地点空间索引

因果辅轴:   E_觉醒仪式 →(A_causal)→ E_同学嫉妒

角色弧光轴: C_莫凡_学徒 →(A_arc)→ C_莫凡_双系法师
```

### 7.3 强关联直接关系

仅以下场景允许实体直接扇出（R_strong）：
- 剧情模块索引内部实体（E_module → C/L/I/R）
- 无明确时序的个人支线事件
- 个人专属武器/功法/物品
- 强关联个体（师徒、父子、挚友）

## 8. 旧关系类型映射

S6 阶段将旧关系类型转换为轴关系类型：

| 旧关系类型 | 新轴关系类型 | 说明 |
|-----------|------------|------|
| `located_in` | `S_topo`（入边） | 空间拓扑归属 |
| `belongs_to` | `S_topo`（入边） | 空间拓扑归属 |
| `participates_in` | `T_branch` | 时序分支（事件挂载到模块） |
| `causes` | `A_causal` | 因果辅轴 |
| `evolves_to` | `evolves_to` | 角色演化（v0.5.0 独立类型，不再映射到 A_arc） |
| `relates_to` | `R_strong` | 强关联（仅保留强关联对） |
| `references` | `R_strong` | 强关联（仅保留专属物品/功法） |

## 9. S6 轴化处理流程

1. **旧关系转换**：按 §8 映射表将旧关系类型转换为轴关系类型
2. **T_main/E_module 审查与重做**（v0.5.1 新增）：调用 `graph_builder.evaluate_skeleton_quality` 评估骨架质量（T_main 卷数 / E_module 颗粒度 / T_branch 覆盖率三维度）。若 verdict=rebuild，调用 `graph_builder.rebuild_skeleton` 用提高后的阈值（0.70/0.85/0.80）重新聚类+构建骨架。重做失败时回退原始骨架
3. **T_main 构建校验**：确认 T_main 仅连接 E_module 实体，每个 E_module 至少有一条 T_main 边（若步骤 2 触发重做，需重新校验）
4. **T_branch 挂载校验**：确认每个 E_event 通过 T_branch 挂载到所属 E_module（若步骤 2 触发重做，需重新挂载）
5. **出边裁剪**：按 §5.2 详表上限裁剪超限实体的出边，保留 strong 优先
6. **S_topo 双向构建**：实体→L 入边（地理归属）+ L→E_module/E_event 出边（空间索引）
7. **关系边索引化**：校验关系 description ≤50 字推荐，>100 字拒绝入库（详见 §6）
8. **扇入补全**：确保被裁剪的实体通过轴索引可被扇入访问
9. **零散节点处理**：孤儿节点和微型节点聚类合并到轴上
10. **人物弧光链构建**（v0.5.0）：调用 `graph_builder.build_evolves_to_relations` 为同一角色在不同卷的档案创建 evolves_to 关系链
11. **人物小传构建**（v0.5.1 新增）：调用 `character_synthesizer.prepare_character_synthesis` 收集各卷角色档案 → LLM 按 5 段结构生成人物小传 → 调用 `character_synthesizer.create_synthesis_entity` 创建 `C_{name}__synthesis` 实体写入 DB
12. **冲突深度校验**：对 `conflict_detected=true` 的记录执行深度图遍历消解

---

## 10. 关系边质量评估

| 质量等级 | 标准 | 处理 |
|---------|------|------|
| **高质量** | T_main（主轴）+ strong + 主线推进 | 保留 |
| **中质量** | T_branch/A_causal/A_arc + strong + 支线发展 | 保留 |
| **低质量** | R_strong + weak + 日常互动 | 可降级为索引 |
| **冗余** | 同类型重复 + 连续无分叉 | 合并或索引化 |
| **详情分散** | description > 50 字 | 触发 LLM 重新归纳 |
| **详情过载** | description > 100 字 | 拒绝入库，强制重抽 |

---

## 11. 零散节点处理

| 节点类型 | 判定标准 | 处理方式 |
|---------|---------|---------|
| **孤儿节点** | 无任何关系边 | 合并到轴上或删除 |
| **微型节点** | 入边 ≤ 1 且 importance ≤ low | 聚类到相邻核心节点 |
| **幽灵节点** | 命名冲突产生的孤立节点 | 合并到正确节点 |
| **冗余节点** | 与已有节点信息高度重复 | 合并去重 |
| **碎片事件** | 单章节原子动作级事件 | 合并到所属跨章节事件单元 |

## §人物小传构建指导（v0.5.1 B2 新增）

### 目的

在 S6 步骤 9 中，为出现在 ≥2 个 T_main 卷的角色构建统合性人物小传。小传是基于各卷档案的综合概括，提供角色的全局视角。

### 输入

- 角色名
- 各卷档案列表（按卷索引排序）: `[{"vol_idx": 0, "info": "..."}, {"vol_idx": 1, "info": "..."}, ...]`

### 输出格式

人物小传必须按以下 5 段结构输出，每段以段首标记开头：

```
【身份概述】
（角色基本身份、背景、定位，≥150 字）

【性格演变】
（性格随卷数的演变轨迹，标注关键转折，≥150 字）

【能力成长】
（能力体系的成长路径，标注里程碑，≥150 字）

【关系网络】
（核心人际关系及变化，标注重要关系，≥150 字）

【人物弧光】
（完整人物弧光总结，串联各卷发展，≥200 字）
```

总字数: 800-2000 字

### 生成要求
